📌 What is this project?

An internet banking backend system built with a microservices architecture. It demonstrates enterprise-grade patterns including OAuth2 security, service discovery, centralized configuration, inter-service communication, and distributed tracing — all running in Docker.

✅ What it does

  • User registration with Keycloak-backed identity management and admin approval workflow
  • Fund transfers between bank accounts with fully atomic debit/credit operations
  • Utility bill payments routed through the core banking engine
  • JWT-protected API with a single gateway entry point
  • End-to-end distributed tracing across all services via Zipkin

💡 What I'd improve next

  • Add event-driven messaging (Kafka) for async transaction notifications
  • Implement Saga pattern with compensating transactions for distributed failures
  • Add circuit breakers with Resilience4j to handle downstream service outages
  • Build a notification service for email/SMS alerts on transactions
  • Add retry logic and dead-letter queues for failed inter-service calls

🎤 Interview Answer (2–3 minutes)

For a standard "tell me about a project" question

"This is an internet banking backend I built to demonstrate real-world microservices design patterns. The system is split into seven services: an API Gateway, a Service Registry, a Config Server, and four business services — User, Core Banking, Fund Transfer, and Utility Payment.

All client requests go through the API Gateway, which enforces OAuth2 security using Keycloak. It validates JWT tokens and forwards the authenticated user's ID to downstream services via a request header.

The Core Banking Service is the heart of the system — it owns all account and transaction data in MySQL. When a fund transfer is initiated, the Fund Transfer Service acts as an orchestrator: it saves the request locally first, then calls Core Banking to execute the actual debit and credit atomically. If anything fails mid-way, the entire database transaction rolls back.

For service discovery I used Eureka, so services find each other by name rather than hardcoded URLs. Configuration is managed centrally by a Spring Cloud Config Server pulling from a Git repository. And I integrated Zipkin with Micrometer for distributed tracing across all services.

If I were to extend this, I'd add Kafka for async event-driven notifications and a Saga pattern with compensating transactions to handle partial failures across services."

🧭 Technical Deep-Dive (5+ minutes)

For a demo or senior engineer audience

"Let me walk you through the architecture layer by layer.

Entry Point — API Gateway. Every request hits the Spring Cloud Gateway first. It validates JWT tokens by calling Keycloak's JWK Set URI. Once verified, it extracts the user ID from the token and passes it downstream via an X-Auth-Id header. Only the user registration endpoint is publicly accessible — everything else requires a valid token.

Authentication — Keycloak. I chose Keycloak as the OAuth2/OIDC provider because it handles the full complexity of token issuance, refresh, and revocation. User registration is a two-step flow: the User Service validates the national ID against pre-seeded Core Banking data, then creates a disabled Keycloak account. An admin must approve it before the user can log in.

Core Banking Service. This is the single source of truth for all financial data. The fund transfer is a fully atomic operation — I fetch both accounts, validate the balance, then debit and credit inside a single @Transactional method. Either both writes succeed, or neither does.

Orchestrator Services. The Fund Transfer and Utility Payment services follow an orchestrator pattern. They record the request locally with a PENDING status before calling Core Banking. On success they update the local record. This gives a full local audit trail even if Core Banking responds slowly.

Observability. Zipkin and Micrometer give me end-to-end trace IDs across all services. I can see exactly where latency comes from in a full request chain — which is critical when debugging distributed systems."

👤 User Registration Flow

  1. User submits registration request with email, password, and national ID.
  2. User Service checks Keycloak — email must not already exist.
  3. User Service calls Core Banking: GET /api/v1/user/{nationalId} to fetch pre-seeded banking record.
  4. Validates that the submitted email matches the core banking record.
  5. Creates a disabled Keycloak user with the given credentials.
  6. Saves a local user record with status PENDING.
  7. Admin approves the user — status becomes APPROVED, Keycloak account is enabled.
  8. User can now log in and obtain a JWT from Keycloak.

💸 Fund Transfer Flow

  1. Authenticated user sends transfer request through the API Gateway.
  2. Gateway validates JWT, extracts user ID, forwards request with X-Auth-Id header.
  3. Fund Transfer Service saves the request with status PENDING.
  4. Calls Core Banking via Feign: POST /api/v1/transaction/fund-transfer.
  5. Core Banking fetches source & destination accounts from MySQL.
  6. Validates source account has sufficient available balance.
  7. Inside one @Transactional block — debits source, credits destination, records both transactions.
  8. Returns a transaction reference ID to the Fund Transfer Service.
  9. Fund Transfer Service updates local record to SUCCESS with the reference ID.

🧾 Utility Payment Flow

  1. User submits a bill payment request (account number + utility provider + amount).
  2. Utility Payment Service saves the request with status PROCESSING.
  3. Calls Core Banking via Feign: POST /api/v1/transaction/util-payment.
  4. Core Banking validates the payer's account and balance.
  5. Deducts the amount atomically and records a UTILITY_PAYMENT transaction.
  6. Utility Payment Service updates local record to SUCCESS with transaction ID.

🚀 Service Startup Order

  1. Zipkin — observability must be up first.
  2. PostgreSQL — needed by Keycloak.
  3. MySQL — needed by all business services.
  4. Keycloak — auth provider must be ready before any service validates tokens.
  5. Config Server — all services pull config on boot.
  6. Service Registry (Eureka) — services register here.
  7. API Gateway & business services — can now start in any order.

❓ Likely Interview Questions

Why did you split it into these specific services? ▼

Each service maps to a clear business domain — user management, core banking, and payments. This lets teams work independently and deploy without coordinating releases. It also lets you scale only what needs it — if fund transfers spike, I scale just that service without touching the others.

How do you ensure data consistency across services? ▼

Within a single service I use @Transactional for ACID guarantees — the debit and credit in Core Banking are one atomic operation. Across services I rely on the orchestrator pattern: the Fund Transfer Service records the request before calling Core Banking, giving a local audit trail. In a production system I'd add a Saga pattern with compensating transactions to handle partial failures.

Why Keycloak instead of rolling your own auth? ▼

Keycloak is battle-tested and handles the full complexity of OAuth2/OIDC — token issuance, refresh, revocation, and user federation. It would be a mistake to build that from scratch. The business value is in the banking operations, not the auth infrastructure. Keycloak also gives me an admin UI for user management out of the box.

How do services find each other? ▼

Through Eureka service registry. Each service registers its host and port on startup. Feign clients resolve service names to actual instances at runtime — I never hardcode URLs. This also enables client-side load balancing automatically when multiple instances of a service are running.

What happens if Core Banking is down when a transfer is requested? ▼

Currently the Feign call fails and the local PENDING record stays in that state — the transfer doesn't go through. In a production system I'd add a circuit breaker with Resilience4j to fail fast and return a clean error to the user, plus a retry mechanism or async queue so the transfer can be retried when Core Banking recovers.

How does distributed tracing work here? ▼

Micrometer with Brave auto-instruments every incoming and outgoing HTTP call. Each request gets a unique trace ID that is propagated via HTTP headers across all service hops. Zipkin collects these spans and lets me visualize the full journey of a request — I can see exactly how long each service took and where errors occurred.

What is the X-Auth-Id header for? ▼

The API Gateway validates the JWT and extracts the Keycloak user ID from the token claims. Rather than making every downstream service validate the token independently, the gateway passes the verified identity forward via the X-Auth-Id header. Downstream services trust this header because they are only reachable through the gateway.
