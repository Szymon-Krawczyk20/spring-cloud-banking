Step 1

Service Registry

internet-banking-service-registry · Port 8081

What Is It?

A Netflix Eureka Server — it acts as a phone book for all microservices. Every other service registers itself here, and services use it to find each other by name instead of hardcoded IPs.

build.gradle — Dependencies

// The single key dependency
implementation 'org.springframework.cloud:spring-cloud-starter-netflix-eureka-server'
implementation 'org.springframework.boot:spring-boot-starter-web'
implementation 'org.springframework.boot:spring-boot-starter-actuator'

Main Class

@EnableEurekaServer        // turns this app into a Eureka Server
@SpringBootApplication
public class InternetBankingServiceRegistryApplication {
    public static void main(String[] args) {
        SpringApplication.run(InternetBankingServiceRegistryApplication.class, args);
    }
}
One annotation — @EnableEurekaServer — is all the Java code needed.

application.yml

server:
  port: 8081

eureka:
  client:
    service-url:
      defaultZone: http://localhost:${server.port}/eureka
    register-with-eureka: false   # don't register itself
    fetch-registry: false          # don't fetch from itself
  instance:
    prefer-ip-address: true
    hostname: localhost

How Other Services Connect

# Every client service has this:
eureka:
  client:
    service-url:
      defaultZone: http://internet-banking-service-registry:8081/eureka
    register-with-eureka: true   # opposite of server
    fetch-registry: true

How to Run Locally

cd internet-banking-service-registry
./gradlew bootRun
# Open: http://localhost:8081  → Eureka dashboard

Summary

FilePurpose
build.gradleAdds Eureka Server dependency
application.ymlConfigures port & disables self-registration
Main class@EnableEurekaServer — that's it
DockerfilePackages jar for Docker with Java 21

The big concept: This service is a "where is everyone?" registry. Every other microservice announces itself here on startup, and when service A needs to call service B, it asks Eureka for B's address instead of hardcoding it.

Step 2

Config Server

internet-banking-config-server · Port 8090

What Is It?

A Spring Cloud Config Server — it serves configuration files (.yml) to all other microservices from a central Git repository. Instead of each service having its own application.yml with database URLs and ports, they all fetch config from this server at startup.

Think of it as: one place to manage config for all services.

build.gradle — Key Dependency

implementation 'org.springframework.cloud:spring-cloud-config-server'

Main Class

@EnableConfigServer       // turns this app into a Config Server
@SpringBootApplication
public class InternetBankingConfigServerApplication { ... }

application.yml — The Important One

server:
  port: 8090

spring:
  cloud:
    config:
      server:
        git:
          uri: https://github.com/JavatoDev-com/internet-banking-microservices-configurations.git
          search-paths: configuration   # look inside this folder
          default-label: main            # use the 'main' branch

How Config Files Are Named in the Git Repo

configuration/
  internet-banking-user-service.yml
  internet-banking-fund-transfer-service.yml
  internet-banking-api-gateway.yml
  internet-banking-user-service-docker.yml   ← docker profile override

How Other Services Consume This

spring:
  config:
    import: configserver:http://internet-banking-config-server:8090
  application:
    name: internet-banking-user-service  # must match filename in config repo

Startup Order (Important!)

1. Service Registry (Eureka) → port 8081
2. Config Server → port 8090
3. All other services → they need config server to be up first

The wait-for-it.sh script in Docker handles this — it waits for dependencies to be ready before starting.

Summary

FilePurpose
build.gradleAdds Config Server dependency
application.ymlPoints to GitHub repo where config files live
Main class@EnableConfigServer — that's it
DockerfileRuns with docker profile active

The big concept: No service owns its own full config. They all ask the config server at startup — "give me my config". Change a DB URL in Git and restart a service, without rebuilding the jar.

Step 3

API Gateway

internet-banking-api-gateway · Port 8082

What Is It?

The single entry point for all client requests. No client talks directly to any microservice — everything goes through the gateway first. It handles:

  1. Routing — forward requests to the right service
  2. Authentication — validate JWT tokens via Keycloak
  3. Header injection — pass the authenticated user's ID to downstream services

build.gradle — Key Dependencies

// Core gateway (reactive)
implementation 'org.springframework.cloud:spring-cloud-starter-gateway'
implementation 'org.springframework.boot:spring-boot-starter-webflux'

// Security - OAuth2/JWT
implementation 'org.springframework.boot:spring-boot-starter-security'
implementation 'org.springframework.boot:spring-boot-starter-oauth2-resource-server'

// Distributed tracing
implementation 'io.micrometer:micrometer-tracing-bridge-brave'
implementation 'io.zipkin.reporter2:zipkin-reporter-brave'
Note: Uses WebFlux (reactive) — not traditional Spring MVC. That's why you'll see Mono, Flux, and ServerHttpSecurity instead of regular HttpSecurity.

bootstrap.yml / bootstrap-docker.yml

# bootstrap.yml — local dev
spring:
  cloud:
    config:
      uri: http://localhost:8090

# bootstrap-docker.yml — Docker
spring:
  cloud:
    config:
      uri: http://internet-banking-config-server:8090

bootstrap.yml loads before application.yml — it tells Spring where to fetch the rest of the config.

SecurityConfiguration.java — JWT Authentication

@EnableWebFluxSecurity   // reactive security
public class SecurityConfiguration {

    @Bean
    public SecurityWebFilterChain securityWebFilterChain(ServerHttpSecurity http) {
        http.authorizeExchange(exchanges -> {

            // Public endpoints (no token needed)
            exchanges.pathMatchers("/user/api/v1/bank-users/register").permitAll();
            exchanges.pathMatchers("/actuator/**").permitAll();

            // Everything else requires a valid JWT
            .anyExchange().authenticated();
        });

        http.csrf(disable);

        // Validate JWT using Keycloak's public key (JWK URI)
        http.oauth2ResourceServer(server ->
            server.jwt(jwt -> jwt.jwkSetUri(jwkUri))
        );
    }
}

How JWT Validation Works (Step by Step)

  1. Client logs in to Keycloak → gets a JWT token
  2. Client sends request with Authorization: Bearer <token>
  3. Gateway calls Keycloak's jwk-set-uri to get the public key
  4. Gateway verifies the token signature using that key
  5. If valid → forward request. If invalid → 401 Unauthorized

GatewayConfiguration.java — Global Filter

@Bean
public GlobalFilter customGlobalFilter() {
    return (exchange, chain) ->
        exchange.getPrincipal()
            .map(Principal::getName)
            .defaultIfEmpty("SYSTEM USER")
            .map(principal -> {
                // Inject username into request header for downstream services
                exchange.getRequest().mutate()
                    .header("X-Auth-Id", principal)
                    .build();
                return exchange;
            })
            .flatMap(chain::filter);
}

Runs on every request. Extracts the authenticated username from JWT and injects it as X-Auth-Id header. Downstream services just read this header — no JWT re-validation needed.

Full Request Flow

Client
│ POST /user/api/v1/bank-users/register
│ Authorization: Bearer eyJhbG...
▼
API Gateway (:8082)
├─ SecurityConfiguration → validate JWT with Keycloak
├─ GatewayConfiguration → inject X-Auth-Id header
│ (looks up "internet-banking-user-service" in Eureka)
▼
User Service (:8083)

Summary

Step 4

Core Banking Service

core-banking-service · Port 8092

What Is It?

The backbone of the entire system — a dummy banking core that manages accounts, balances, fund transfers, and utility payments. All other services call this via OpenFeign to do actual banking operations.

Entity → DTO Pattern

BankAccountEntity (DB table)
│
BankAccountMapper.convertToDto()
│
BankAccount (DTO sent in API response)

You never expose raw DB entities to callers — always convert to DTO first.

AccountService.java

public BankAccount readBankAccount(String accountNumber) {
    BankAccountEntity entity = bankAccountRepository
        .findByNumber(accountNumber)
        .orElseThrow(EntityNotFoundException::new);  // 404 if not found
    return bankAccountMapper.convertToDto(entity);
}

TransactionService.java — Core Business Logic

Fund Transfer

@Transactional   // if anything fails, entire operation rolls back
public FundTransferResponse fundTransfer(FundTransferRequest request) {
    BankAccount from = accountService.readBankAccount(request.getFromAccount());
    BankAccount to   = accountService.readBankAccount(request.getToAccount());

    validateBalance(from, request.getAmount());  // check sufficient funds

    String transactionId = internalFundTransfer(from, to, request.getAmount());
    return FundTransferResponse.builder()
        .message("Transaction successfully completed")
        .transactionId(transactionId).build();
}

internalFundTransfer() — What Actually Happens

// 1. Debit from sender
fromAccount.setActualBalance(fromAccount.getActualBalance().subtract(amount));
bankAccountRepository.save(fromAccount);

// 2. Save DEBIT transaction record (amount is negative)
transactionRepository.save(TransactionEntity.builder()
    .transactionType(FUND_TRANSFER)
    .amount(amount.negate())   // negative = money going out
    .account(fromAccount)
    .transactionId(transactionId).build());

// 3. Credit receiver
toAccount.setActualBalance(toAccount.getActualBalance().add(amount));
bankAccountRepository.save(toAccount);

// 4. Save CREDIT transaction record (amount is positive)
transactionRepository.save(TransactionEntity.builder()
    .transactionType(FUND_TRANSFER)
    .amount(amount)            // positive = money coming in
    .account(toAccount)
    .transactionId(transactionId).build());
Both transaction records share the same transactionId — that's how you trace a transfer end-to-end. The whole method is @Transactional — if anything fails, everything rolls back. No partial transfers.

GlobalExceptionHandler.java

@ControllerAdvice  // applies to ALL controllers
public class GlobalExceptionHandler {

    @ExceptionHandler(SimpleBankingGlobalException.class)
    protected ResponseEntity handleGlobalException(...) {
        return ResponseEntity.badRequest()
            .body(ErrorResponse.builder()
                .code(exception.getCode())
                .message(exception.getMessage()).build());
    }
}
